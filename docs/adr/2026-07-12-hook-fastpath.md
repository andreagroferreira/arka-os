# ADR: Node fast-path shims for PreToolUse/PostToolUse (F2-6)

**Date:** 2026-07-12
**Status:** Accepted
**Context:** ruflo teardown campaign, Frente 2 (Claude Code integration).

## Problem

PreToolUse and PostToolUse fire on EVERY tool call and cost 82ms/96ms
p50 each (bash spawn + interpreter resolver + Python import chain —
`benchmarks/results.md`, baseline 2026-07-12). A typical turn makes
10-20 tool calls: 1.8-3.5s of pure hook overhead. The overwhelming
majority of those calls are trivially allowed (Read/Grep/Glob, discovery
Bash, benign MCP calls) — they pay the full chain to reach a decision
that needs none of it.

## Decision

Two Node shims (`config/hooks/pre-tool-use.cjs`, `post-tool-use.cjs`)
registered as the hook commands on POSIX, backed by a pure decision
engine (`config/hooks/_lib/fastpath/engine.cjs`) and a generated
manifest (`config/hooks/gate-manifest.json`).

**Security invariant #0: the shim has no deny path.** It fast-allows a
closed, corpus-proven set or delegates to the sibling `.sh` (the exact
current chain). Any deny always comes from Python. A shim bug degrades
to latency or to today's behavior — never to a skipped deny. Structural
test: `engine.cjs` cannot contain a deny action or a
`permissionDecision` payload (grep-source test, precedent
`hook-lib.test.js`).

**Manifest, not constants.** Every value the engine consumes is
generated by `core/hooks/gate_manifest.py` from the REAL gate constants
(`python -m core.hooks.gate_manifest`), the same pattern as
`docs_stats.py`. `tests/python/test_gate_manifest_parity.py` fails on
byte drift generator↔committed and executes the embedded corpora against
the real Python functions; `tests/installer/gate-manifest.test.js` runs
the SAME corpora against the engine. Drift on either side breaks the
build.

### Fast paths (everything else delegates)

| Path | Condition | Parity duty replicated |
|---|---|---|
| Pre P4 | tool ∉ flow-gated ∧ ∉ research set | kb_first.jsonl append (`tool-not-gated`) |
| Pre P5 | Bash ∧ discovery (`bash_is_effect` replica) ∧ no active budget cap | kb_first + enforcement.jsonl appends |
| Post Q6 | benign exit ∧ no error-trigger ∧ tool ∉ {ExitPlanMode, Task, Agent} ∧ (hardEnforcement off ∨ flow-auth fresh) | mcp-usage.jsonl append for `mcp__*`, stdout `{}` |

Key semantic pins (all corpus-tested):

- `bash_is_effect` replica: blacklist-first, `VAR=` prefix strip,
  whitelist first token, **unknown → effect → delegate** (a
  misclassification can only cost latency, not a skipped deny — the
  Python side re-classifies after delegation).
- Python truthiness for `hooks.hardEnforcement` (`"false"` is ON, `[]`
  and `{}` are OFF — `engine.pythonTruthy`, where JS `Boolean()`
  disagrees).
- Budget semantics mirror `cost_governor` exactly: missing/corrupt
  config → `{}` → no-cap (cost_governor.py:68-86); a cap is active only
  when `float(value) > 0`. The design draft said "corrupt config →
  delegate"; disk truth says Python fails open deterministically, so
  the shim mirrors it (verified by
  `test_budget_semantics_match_cost_governor`).
- Strict session-id allowlist incl. dot-only rejection
  (`core/shared/safe_session_id.py`) before any auth-file path is built.

### Q6 and the flow-auth confirm rescan

Python's PostToolUse re-confirms flow authorization from the transcript
on every call. The shim skips that rescan only when it is provably
redundant: hardEnforcement off (the pre-side flag check short-circuits
before ever reading auth) or a confirmed auth within TTL. The skipped
refresh self-heals: when the auth ages past TTL, Q7 delegates, Python
rescans and re-confirms, and Q6 resumes. Worst case is one delegated
call every 12h; flipping hardEnforcement ON mid-session behaves the
same way (first stale-auth call delegates and re-confirms).

### Telemetry parity

Fast-path appends replicate the exact serializer output (templates
generated from the real dataclasses; ts is Python-isoformat-parseable,
ms padded to 6 digits). On delegation the shim writes NOTHING — Python
owns the line; no double-log by construction. Concurrency: single JSON
lines « 4KB via `O_APPEND` are atomic on POSIX; Python's `flock` and
the shim's raw append can interleave only at line granularity, which
JSONL readers already tolerate (`corrupt_line_count`). The complex
stores (gotchas, metrics, patterns — lock+replace) are never touched by
the shim: always delegated.

### Wiring and degradation ladder

- Deploy: `installer/hook-lib.js::copyHookAssets` (single asset list,
  shared by index.js and update.js — the v4.3.2 two-loop drift lesson);
  `engine.cjs` travels inside `_lib/` via the existing `copyHookLib`.
- Registration: `hookEntry` registers `<name>.cjs` on POSIX only when
  the deployed file exists; `ARKA_INSTALL_FASTPATH=0` opts out at
  install time. Windows keeps `.ps1` → Python untouched (the `.cjs`
  stdin contract is only validated for bash-spawned hooks).
- Runtime kill switch: `ARKA_HOOK_FASTPATH=0` → the shim delegates
  unconditionally and re-exports the var to the child (recursion guard).
- Missing manifest / corrupt stdin / any internal error → delegate;
  missing sibling `.sh` → fail-open with the per-event degraded form
  (Pre: empty stdout exit 0; Post: `{}` exit 0).
- Node removed post-install → the hook command exits 127; Claude Code
  treats non-2 exits as non-blocking → fail-open. `arkaos doctor`
  check `hook-fastpath` FAILs with the remediation.

## Measured result (benchmarks/hooks-bench.sh, N=20, isolated HOME)

pre-tool-use 83→18ms p50 (−78%), post-tool-use 99→19ms (−81%).
Delegated calls pay +18ms (node startup). Net for a typical 15-call
turn ≈ −600ms. The residual 18ms is the Node startup floor on the
reference machine (`node -e ''` ≈ 10ms); a sub-15ms fast path would
require a resident process, which the F2 plan explicitly rejected
(the MCP server is the resident process where residency pays).

## Rejected alternatives

- **Queue + drainer for PostToolUse** (reader 2's draft): deferred
  replay breaks point-in-time semantics (`git branch`, workflow-state)
  and adds a second writer to canonical stores.
- **Replicating gate logic beyond the proven set** (specialist globs,
  frontend markers, grace accounting): every one of those paths writes
  decision-bearing state; the deadlock regression #297 came from
  exactly such an omission. Delegation is correct there by design.
- **Resident Python daemon**: rejected in the F2 plan — process-count,
  not cold-start, dominates; the arka-tools MCP server (F2-3) is the
  resident surface.
